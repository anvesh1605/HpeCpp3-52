# Resolved Issues

This section describes the issues resolved in this release.

| Category | Bug ID | Description |
| --- | --- | --- |
| IPsec | T‐5798 | **Symptom:** IPsec packets not intended to the DSS getting dropped when edge services mode is not enabled.  **Scenario:** When IPsec packets are exchanged between a server‐client with DSS in between, DSS is expected to forward the packet without any modification as the destination is not intended to the DSS. But when **edge\_services** mode is not enabled, DSS might drop the packet instead of forwarding to the destination.  **Workaround:** Enable edge services mode to make DSS aware of IPsec transit packets. |
| Policies | T‐5917 | **Symptom:** Sflow generates an excessive number of logs.  **Scenario:** When a Policy Rule is configured with protocol set to any and application is configured. Traffic matching this Rule can become unpredictable.  **Workaround:** Create 2 instances of the Policy rules, one with the desired matching protocols and another one with the desired matching application. |
| Counters | 352501 | **Symptom:** Switches observe a crash after applying specific configuration with the switchd\_agent error.  **Scenario:** In some scenarios after configuring Internet Protocol Flow Information Export (IPFIX) daemon switchd\_agent crash may crash. |
| REST | 348886 | **Symptom:** The webserver becomes unresponsive, but AFC is reachable. Restarting HTTPS and Nginx services does not recover connectivity. Netstat for ports 443 and 80 do not work as expected.  **Scenario:** In some scenarios where there is an excessive memory usage and resource exhaustion, the nginx process is incorrectly terminated by the switch leading to webservice to become unresponsive. |
| Multicast | 348640 | **Symptom:** Unexpected multicast flooding on VLAN after upgrading from 10.10 to 10.13.  **Scenario:** In some scenarios when upgrading from 10.10 to 10.13, IGMP flooding can be observed for VLANs that have IP IGMP snooping enabled.  **Workaround:** Disable and re‐enable IP IGMP snooping on affected VLAN. |
| Port Access | 347667 | **Symptom:** Some mac‐auth clients show incorrect VLAN mapping after being authenticated. The **show mac‐address table detail** command displays the default unauthenticated VLAN instead of the authorized VLAN.  **Scenario:** In some cases during client authentication, client MAC is not aged out correctly after the client has been successfully authenticated and placed in authenticated VLAN. |
| sFlow | 347443 | **Symptom:** sFlow generates an excessive number of logs.  **Scenario:** When sFlow is enabled in the switch an excessive number of sFlow irrelevant logs are generated. these logs could be throttled. |
| MSTP | 346938 | **Symptom:** Intermittent MSTP/RPVST starvation messages observed coming from source mac address 00:00:00:00:00:00.  **Scenario:** When 3rd party legacy devices send packets with source mac address 00:00:00:00:00:00 to CX devices intermittent BPDU starvation might be observed.  **Workaround:** Configure ACL to drop invalid source mac address 00:00:00:00:00:00 in all switches where end devices are connected. |
| Central | 344467 | **Symptom:** The **show aruba‐central** command displays the switch as connected but Central WebUI shows the **connection failure** error.  **Scenario:** In some scenarios, the internal process in charge of initialize the connection to Central is not responding therefore the connection is refused in the Central WebUI. |
| Port Security | 343889 | **Symptom:** The sticky‐learn clients are not shown post reboot.  **Scenario:** When port is down and switch is rebooted, entry is lost, and sticky‐learn clients will not be shown. |
| Diagnostics | 343517 | **Symptom:** Cable diagnostics test does not return cable distance.  **Scenario:** In some scenarios, cable diagnostic test feature is unable to return cable distance for pair of cables that are open. |
| Mirroring | 343456 | **Symptom:** ERSPAN session not active and the show mirror <id> command display status field as **external/driver\_error**.  **Scenario:** When mirror destination endpoint is configured with sub‐interface, the mirror session will not be active. |
| PBR | 343095 | **Symptom:** Traffic loss observed for traffic matching with configured PBR policies in a VSX setup.  **Scenario:** In some VSX configurations with PBR policies where the next hop is learned over MCLAGs, traffic loss can be observed after local links go down and new next hop is learned on different MCLAG.  **Workaround:** Clear ARP. |
| ARP | 342578 | **Symptom:** Proxy ARP and Local Proxy ARP functionality is not available. |
| WebUI | 342578 | **Symptom:** Warning messages displayed for FAN tray status in the WebUI monitoring tool, despite having only one FAN tray installed. The switches were otherwise operating within normal parameters.  **Scenario:** Single fan tray when monitoring via the WebUI. The issue was prominent in scenarios where only one FAN tray was connected. |
| Mirroring | 342437 | **Symptom:** Packets near MTU limits in size may not be present in a port or tunnel/remote based mirror.  **Scenario:** When a client configures a mirror where the destination interface has the same MTU size, mirrored packets that are close to max MTU are incorrectly dropped and not mirrored. This applies to remote/tunnel mirrors as well.  **Workaround:** Configuring mirror destination ports as routing port and set the **IP MTU** size. For a VLAN interface, set the IP MTU on the VLAN SVI. IP MTU must be set to accommodate the ethernet overhead +tag. For the default MTU, 1500, the IP MTU would be 1518. For a tunneled scenario, the MTU must also account for the tunnel header. |
| PTP | 341991 | **Symptom:** PTP may not work after reboot of a ptp‐unaware neighbor device on setups with a large number of PTP clients.  **Scenario:** This is issue is observed when the default PTP timers are modified and the device has a large number of PTP clients configured (over 20 per port).  **Workaround:** For large scale PTP deployments, keep the PTP timers set to default. |
| ACL Logging | 341769 | **Symptom:** Incorrect ACL seq# is applied in control plane ACL's.  **Scenario:** When an ACLs with both comments and the log action are applied to the control plane, incorrect ACL seq# is configured for ACE.  **Workaround:** Remove comments the ACL from the control plane. |
| VSX | 341536 | **Symptom:** Traffic does not flow over MCLAGs, causing the Egress Block to remain unset. This results in traffic from the ISL being sent out over the MCLAG, potentially causing connected devices to receive duplicate packets.  **Scenario:** Configuring an MCAG with AFC.  **Workaround:** Create a new VLAN on the devices where the MCLAG was configured. |
| VSX | 341536 | **Symptom:** Traffic does not flow over MCLAGs, causing the Egress Block to remain unset. This results in traffic from the ISL being sent out over the MCLAG, potentially causing connected devices to receive duplicate packets.  **Scenario:** Configuring an MCAG with AFC.  **Workaround:** Create a new VLAN on the devices where the MCLAG was configured. |

Setup the client MAC that is trying to onboard such that the MAC is denied. For example: Either by failing authentication or due to max client limit. Move such a client to a non‐secure port. Observe the MAC table that the MAC is not moved to the non‐secure port. |
| ARP | 340677 | **Symptom:** Traffic loss and ping failure due to ndmd crash.  **Scenario:** Performing the **clear arp** command with a short interval.m |

| VXLAN | 339717 | **Symptom:** ops‐switchd crash while collecting VXLAN diagdump or collecting support files all.  **Scenario:** In some VXLAN scenarios with multiple LAGs and multiple overlay VLAN on those lags. After device reboot, the LAG interface could show incorrect VXLAN Programming. If the user attempts to remove VLANs, VNI‐to‐VXLAN configurations that belong to the affected LAGS, a ops‐switchd crash might be observed.  **Workaround:** Ensure LAG interfaces are up before configuring the VLANs for the lag. |
| REST | 338341 | **Symptom:** Configuration push from Central fails due to a Dry Run error with the message **systemNotReady**.  **Scenario:** In some scenarios, the internal Dry Run state may become invalid leading to failing when pushing the configuration. |
| VXLAN | 337743 | **Symptom:** VXLAN packets with UDP source port 69 are dropped at the wireless gateway.  **Scenario:** Occurs when stub switch sends VXLAN packets with well‐known ports instead of RFC7438 recommended range.  **Workaround:** Move user traffic from vxlan vlan to a non‐vxlan vlan. |
